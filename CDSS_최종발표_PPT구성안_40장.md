### 슬라이드 4: 프로젝트 배경 및 문제 정의
**제목**: 왜 CDSS가 필요한가?

**현재 의료 시스템의 문제점**:
- ❌ **의료진 업무 과부하**: 진단 시간 부족, 업무 효율성 저하
- ❌ **데이터 분산 관리**: EMR, OCS, RIS, LIS, PACS 시스템 간 연동 부족
- ❌ **주관적 진단**: 의료진 경험에 의존한 진단, 일관성 부족
- ❌ **조기 진단의 어려움**: 환자 접근성 제한, 스크리닝 기회 부족

**우리 프로젝트의 해결 방안**:
- ✅ **AI 기반 진단 지원**: 멀티모달 AI 분석으로 정확도 향상
- ✅ **통합 의료 정보 시스템**: EMR, OCS, RIS, LIS, PACS 통합 관리
- ✅ **객관적 진단 기준**: AI 분석 결과 기반 일관된 진단 지원
- ✅ **역할별 맞춤형 인터페이스**: 6가지 역할별 최적화된 시스템

**시각 자료**:
- 문제점 vs 해결 방안 비교표
- Before/After 다이어그램

### 슬라이드 8: 데이터베이스 설계
**제목**: 데이터베이스 설계

**주요 테이블**:
1. **환자 관리**
   - `PatientUser`: 환자 계정 정보
   - `Patient`: 환자 기본 정보
   - `Appointment`: 예약 정보

2. **의료진 관리**
   - `User`: 의료진 계정 (Django 기본)
   - `Doctor`: 의사 정보
   - `Department`: 진료과 정보

3. **OCS (처방전달시스템)**
   - `Order`: 주문 정보 (처방/검사/영상)
   - `OrderItem`: 주문 항목 상세
   - `DrugInteraction`: 약물 상호작용 정보

4. **의료 이미지**
   - `MedicalImage`: 의료 이미지 메타데이터
   - `ImageAnalysis`: AI 분석 결과

5. **진료 기록**
   - `MedicalRecord`: 진료 기록
   - `Prescription`: 처방전

**시각 자료**:

### 슬라이드 15: 환자 로그인 및 회원가입
**제목**: 환자 로그인 및 회원가입

**Flutter 모바일 앱**:
- 환자 회원가입
- 로그인 (계정 ID + 비밀번호)
- 자동 로그인 (Remember Me)
- 비밀번호 찾기

**웹 홈페이지**:
- 동일한 로그인 시스템
- 반응형 디자인

**인증 프로세스**:

**제목**: 맘모그래피 AI 분석

- **모델**: ResNet50
- **입력**: 4장 맘모그래피 이미지 (L-CC, R-CC, L-MLO, R-MLO)
- **출력**: 4-class 분류 결과
- **클래스**: Mass, Calcification, Architectural/Asymmetry, Normal

```
1. 4장 DICOM 이미지 선택

   ↓
3. Mosec → Orthanc에서 DICOM 다운로드
   ↓
4. 전처리 (Otsu 임계값, Contour 검출, Crop, Resize 512×512)
   ↓
5. ResNet50 모델 추론
   ↓
6. 각 이미지별 분류 결과 반환
   ↓
7. 프론트엔드에 결과 표시
```

**결과 표시**:
- 각 뷰별 예측 클래스
- 확률 (0-100%)
- 클래스별 확률 바 차트

### 슬라이드 29: 병리 이미지 AI 분석
**제목**: 병리 이미지 AI 분석

**모델 정보**:
- **분류 모델**: CLAM (Attention MIL)
- **Feature Extractor**: H-optimus-0 (Bioptimus)
- **세그멘테이션 모델**: nnUNet
- **입력**: WSI (Whole Slide Image, SVS 파일)
- **출력**: Normal/Tumor 분류 + 세그멘테이션 결과
- **서비스**: 교육원 PC 워커 (HTTP API 방식)

**특징**:
- GCP 서버의 Mosec이 아닌 교육원 PC에서 실행
- 로컬 wsi 폴더의 파일 직접 사용
- HTTP API로 요청 폴링 및 결과 전송

**분석 프로세스**:
```
1. SVS 파일 업로드 (Orthanc)
   ↓
2. SVS → DICOM 변환 (Orthanc)
   ↓
3. Django → 교육원 PC 워커 요청 생성 (HTTP API)
   ↓
4. 교육원 PC 워커 → 요청 폴링 (30초마다)
   ↓
5. 교육원 PC 워커 → 로컬 wsi 폴더에서 파일 읽기
   ↓
6. OpenSlide로 WSI 열기
   ↓
7. 패치 추출 (최대 1000개, 224×224)
   ↓
8. Feature 추출 (H-optimus-0)
   ↓
9. CLAM 모델 추론 (Attention MIL) - 분류
   ↓
10. nnUNet 모델 추론 - 세그멘테이션 (선택적)
   ↓
11. 결과를 Django API로 전송 (HTTP POST)
   ↓
12. 프론트엔드에 결과 표시
```

**결과 표시**:
- 분류 결과 (Normal/Tumor)
- 신뢰도 (%)
- 클래스별 확률
- 주요 관심 영역 (Top attention patches)

**시각 자료**:
- 분석 프로세스 플로우차트
- CLAM 모델 구조도
- nnUNet 세그멘테이션 모델 구조도
- 결과 화면 스크린샷

### 슬라이드 30: MRI 세그멘테이션
**제목**: MRI 세그멘테이션

**모델 정보**:
- **모델**: SwinUNETR
- **입력**: 4-ch DCE-MRI (4개 시퀀스, 각 96 슬라이스)
- **출력**: 3D 세그멘테이션 마스크 (DICOM SEG)
- **목적**: 종양 영역 세그멘테이션

**분석 프로세스**:
```

   ↓

   ↓
3. 중앙 96개 슬라이스 선택
   ↓
4. Django → Mosec 요청 (instance_ids만 전송)
   ↓
5. Mosec → Orthanc에서 DICOM 다운로드 (384개 파일)
   ↓
6. 3D 볼륨 변환 ([4, 96, 256, 256])
   ↓
7. 다운샘플링 ([4, 96, 96, 96])
   ↓
8. SwinUNETR 모델 추론 (sliding window)
   ↓
9. 후처리 (임계값 0.7, 형태학적 침식, 작은 객체 필터링)
   ↓
10. DICOM SEG 생성 (96 프레임)
   ↓
11. Orthanc에 업로드
   ↓
12. 프론트엔드에 오버레이 표시
```

**결과 표시**:
- 세그멘테이션 오버레이 (원본 MRI 위에 표시)

### 슬라이드 35: OCS (처방전달시스템) 개요
**제목**: OCS (Order Communication System)

**OCS란?**
- 의사가 처방전, 검사, 영상 촬영을 주문하고 각 부서로 자동 전달하는 시스템

**주문 유형**:
1. **처방전** (Prescription)
   - 약국으로 전달
   - 약물 상호작용 체크
   - 알레르기 체크

2. **검사** (Lab Test)
   - 검사실로 전달
   - LIS 연동

3. **영상 촬영** (Imaging)
   - 방사선과로 전달
   - RIS 연동

4. **조직 검사** (Tissue Exam)
   - 병리과로 전달

**주문 상태**:
- 대기중 (pending) → 전달됨 (sent) → 처리중 (processing) → 완료 (completed)

**시각 자료**:
- OCS 시스템 구조도
- 주문 프로세스 플로우차트

**제목**: OCS 주요 기능

**1. 주문 생성 및 관리**
- 의사가 주문 생성

- 주문 이력 조회

**2. 자동 검증 시스템**
- **약물 상호작용 체크**
  - 두 약물 간 상호작용 검사
  - 심각도 분류 (경미/중등도/심각)
  - 심각한 상호작용 발견 시 주문 전달 불가

- **알레르기 체크**
  - 환자 알레르기 정보와 주문 항목 비교
  - 알레르기 발견 시 경고 표시

**3. 우선순위 관리**
- 일반 (routine)
- 긴급 (urgent)
- 즉시 (stat)
- 응급 (emergency)

**4. 실시간 알림**
- 주문 상태 변경 시 알림
- 부서별 알림

**시각 자료**:
- OCS 화면 스크린샷
